The mind-boggling state of web development

2014-6-8

Have you ever looked back at stuff from the 90s and wondered how the hell people were oblivious to how crappy their technology was? Like giant cell phones, or skipping cd players?

Yeah. One day we'll look back at the year 2014 and wonder how the hell people put up with the terrible state of web development. Specifically the number of awful tools necessary to create anything useful. It's really that bad. Here's what you need to know and use to make a website that does something:

  • HTML if you just want to make a website.
  • CSS if you don't want it to look hideous.
  • JavaScript if you want any moving parts.
  • PHP, Ruby, or Python if you want your website to do something.
  • SQL if you need to store information.

And that's the minimum, just to get a working web application. All of these technologies are complex, all of them are old and archaic. They are. Because they're so difficult to work with, here's the other stuff you'll probably have to learn:

  • Jquery, because JavaScript is terrible with HTML.
  • Coffeescript, because JavaScript is terrible period.
  • HTML5 Canvas if you want moving parts of more complexity.
  • EaseJS or some other canvas library, because Canvas is terrible.
  • Ruby on Rails, Django, or another web framework because server-side development is terrible.
  • Objective C to make an iPhone app, Java to make an Android app. Both are terrible.

I could go on and on. Now the reason why we have so many different technologies is because everything has been hacked together. The whole thing started from HTML, which was never designed to create web documents, it was never designed with the internet in mind at all. It was the first thing we used, and we've stuck with it. Then we kept on adding extra things on top of it, to make it do things it was never designed to do. And the reason why we never changed it is because we need to maintain backwards compatibility - any change to this system would mean all the billions of websites that currently exist wouldn't work anymore.

Well to hell with backwards compatibility. This system needs to change. Primarily by reducing the number of different technologies necessary for web development down to an optimal minimum. There's no reason why the client-side language should be different from the server-side language. That language should be built from the ground up with the needs of web development frameworks in mind. No CSS and JavaScript animation hacks, no separate Canvas for independent animations, it should be designed with that interactivity in mind. Ideally HTML and CSS should be abolished. Content should exist as markdown with standard sections, and CSS should be integrated as an object oriented part of the programming language. Now this makes it difficult for graphic designers or hobbyists to create websites, since it would require them to learn how to program. So there can be graphical tools designed to aid in that process. Such tools already exist, like Weebly, but they can be made more or less customizable. Creating apps shouldn't require separate languages, they should be a subset of the single development language. The goal is simple - one language to rule them all. It may not be 100% possible, but it sure as hell should be the goal.

So what about backwards compatibility? Well, that's pretty simple, browsers should support both the old and new protocol. We'll need a new protocol anyway to replace http, since we won't be transfering HTML files anymore. Oh, and there should be browser validation the same way there is website validation. Every browser needs to render the site in the same way. That's such a no-brainer, yet so much development time is being sucked into browswer compatibility issues. Anyhow. I don't know how feasible all these ideas are, but there needs to be more effort into making this move forward. Our hardware has been marching forward with Moore's law year after year. Our development software...well, it's still stuck in the 90s, up Dawson's Creek without a paddle.
